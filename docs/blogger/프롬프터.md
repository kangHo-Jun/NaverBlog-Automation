# 블로그 프롬프터 가이드

이 문서는 현재 코드에 들어 있는 주요 프롬프트와 전처리 지시를 초보자도 쉽게 수정할 수 있도록 정리한 문서다.

주의:

- `analyzeHtml()` / `analyzeTxtScript()`는 AI 프롬프트 함수가 아니라, 입력 파일을 정리하는 전처리 함수다.
- 실제 Gemini 분석 프롬프트는 `STEP_B_geminiAnalysis()` 안에 있다.
- 아래 문서에서는 사용자가 이해하기 쉽게 전처리 규칙과 실제 AI 프롬프트를 함께 묶어서 설명한다.

---

## Gemini 전처리 프롬프트
📍 위치: `src/Code.gs` / `analyzeHtml()`, `analyzeTxtScript()`, `STEP_B_geminiAnalysis()`

🎯 역할: HTML/TXT 파일에서 핵심 텍스트를 정리하고, Gemini가 요약/아웃라인/팩트를 JSON으로 추출하도록 지시한다.

⚙️ 수정 가능 부분:

- HTML에서 제거할 태그 범위
- TXT 정제 규칙
- Gemini가 반환해야 할 항목 개수와 길이
- SEO 보완 정보 길이
- 개인 경험 표현 변환 규칙

⚠️ 주의사항:

- `summary`, `outline`, `key_facts`, `seo_addition` 키 이름은 유지해야 한다.
- `outline`은 배열이어야 한다.
- JSON 외의 문장이 섞이면 후속 파싱이 깨진다.
- `analyzeHtml()` / `analyzeTxtScript()`는 프롬프트가 아니라 입력 정제 로직이므로, 여기서 텍스트를 너무 많이 잘라내면 Gemini 분석 품질도 떨어진다.

✏️ 수정 예시:

- 핵심 요약 길이를 줄이고 싶을 때
  - `'summary': 참고글 핵심 요약 (150자 내외)`
- SEO 보완을 더 길게 받고 싶을 때
  - `'seo_addition': SEO 키워드 기반 최신 보완 정보 (500자 내외)`
- TXT에서 더 많은 문단을 참고하고 싶을 때
  - `reference_snippets = paragraphs.slice(0, 8)`

### 전처리 규칙 1: `analyzeHtml()`

실제 AI 프롬프트는 없고, 아래 규칙으로 HTML을 정리한다.

```javascript
// 스크립트, 스타일, 광고, 댓글, 사이드바 제거
// 이미지 태그는 [이미지: alt] 또는 [이미지] 로 축약
// 링크는 텍스트만 보존
// 본문 텍스트 최대 5만자 제한
// h2, 문단, 상위 키워드, 문장 길이 등 추출
```

핵심 처리:

- `<script>`, `<style>`, `<noscript>` 제거
- 광고/iframe/댓글/사이드바/aside/footer 제거
- `<img>`는 `[이미지: 설명]` 형태로 축약
- `<a>`는 텍스트만 남김
- 본문 최대 5만자 제한

### 전처리 규칙 2: `analyzeTxtScript()`

실제 AI 프롬프트는 없고, 유튜브 스크립트 성격의 TXT를 아래처럼 정리한다.

```javascript
// [음악], [박수], [Korean (auto-generated)] 제거
// 짧은 줄들을 합쳐 문단화
// 본문 최대 5만자 제한
// 상위 문단 5개, 키워드 8개 추출
```

핵심 처리:

- 자막 메타 텍스트 제거
- 여러 공백 정리
- 짧은 줄을 문단으로 병합
- HTML보다 키워드를 조금 더 많이 추출

### 실제 Gemini 분석 프롬프트 전문

```text
당신은 전문 콘텐츠 분석가입니다.

[원본 글]
{fulltext}

[SEO 키워드]
{seoKeywords}

다음 항목을 추출하여 JSON 형식으로만 반환하세요:
1. 'summary': 참고글 핵심 요약 (300자 내외)
2. 'outline': 구조 아웃라인 (5개 이내, 문자열 배열 형식)
3. 'key_facts': 핵심 수치 및 팩트 (200자 내외)
4. 'seo_addition': SEO 키워드 기반 최신 보완 정보 (300자 내외)

Markdown 예시 없이 순수 JSON 객체만 반환할 것.

※ 개인 경험담/전문가 개인 시점은 반드시 회사/제품 중심 시점으로 변환할 것
예) '제가 경험해보니' -> '현장에서 검증된' / '저는 추천합니다' -> '저희 제품은'
```

## Claude 글쓰기 프롬프트
📍 위치: `src/Code.gs` / `createReconstructedPromptWithTemplate()`

🎯 역할: 전처리된 벤치마킹 자료, 시트2 스타일, 시트3 템플릿, SEO 키워드를 합쳐 최종 블로그 글을 작성하게 한다.

⚙️ 수정 가능 부분:

- 오프닝 후킹 규칙
- AI 금지 패턴
- 인간적 문체 지시
- 제목 생성 규칙
- 사진 플레이스홀더 개수
- CTA 스타일 문장

⚠️ 주의사항:

- `[사진 X: English description]` 형식은 유지해야 한다.
- `# 제목` + 부제목 구조를 유지하는 편이 안전하다.
- 금지 표현을 너무 많이 넣으면 문체가 뻣뻣해질 수 있다.
- 사진 플레이스홀더 최소 개수를 너무 낮추면 본문 비주얼 구성이 약해진다.

✏️ 수정 예시:

- 오프닝을 더 부드럽게 바꾸기
  - `- 첫 문단은 독자의 일상 경험에서 자연스럽게 시작하라`
- AI 금지 패턴 추가
  - `- "이 글에서는"으로 문단 시작 금지`
- 인간적 문체를 더 강화
  - `- 짧은 감탄형 문장 1~2개 자연 삽입`

### system 프롬프트 전문

```text
너는 {templateInfo.name} 콘텐츠 전문 작가다. 주어진 벤치마킹 자료를 참고하여 완전히 새로운 관점과 구조로 글을 재창조한다.{fileTypeNote}

재창조 원칙:
- 벤치마킹 자료: {benchContentsWeight}% 정도만 참고용으로 활용
- 강조 키워드: {myContentsWeight}% 비중으로 핵심 주제화
- 완전 새로운 스토리와 구조로 재구성
- SEO 키워드를 자연스럽게 배치

템플릿 가이드라인:
- 콘텐츠 유형: {templateInfo.name}
- 글쓰기 스타일: {templateInfo.writing_style}
- 구조: {templateInfo.content_structure}
- 강조 영역: {templateInfo.key_focus_areas}
- 어조: {templateInfo.tone_description}

스타일 가이드라인:
{styleDescription}

사진 삽입 원칙:
- 템플릿 특화 사진: {templateInfo.photo_guide_type}
- [사진 X: English description] 형태로 정확히 표기
- 사진 플레이스홀더 설명은 반드시 영어로 작성하라
- 예) [사진1: slim window frame close-up]
- 예) [사진2: modern interior living room window view]
- 예) [사진3: before and after window replacement comparison]
- 전체 글에 최소 8-12개의 사진 플레이스홀더 포함

오프닝 원칙 (매우 중요):
- 첫 문단은 반드시 아래 중 하나로 시작하라:
- 독자의 고통을 찌르는 질문
  예) "창호 바꾸고 나서 '왜 이걸 진작에 안 했지?' 하신 분 계세요?"
- 충격적인 숫자나 사실
  예) "7mm. 이 차이 때문에 수억짜리 조망이 반토막 납니다."
- 공감 유발 상황 묘사
  예) "새벽에 베란다 나갈 때마다 발에 걸리는 그 돌출 부분..."
- 절대 배경 설명이나 주제 소개로 시작하지 마라

인간적 문체 원칙:
- 짧은 문장(10자 이하)과 긴 문장(40자 이상) 혼합
- 독자에게 직접 말하는 2인칭 사용 "여러분", "~하셨나요?"
- 전문가 경험담 1인칭 삽입 "제가 현장에서 보면..."
- 반전 표현 사용 "그런데 말입니다", "사실은..."
- 구어체 자연 삽입 "솔직히", "딱 잘라 말하면"

제목 생성 원칙 (매우 중요):
- 글 맨 앞에 # 제목을 반드시 포함하세요
- 제목 길이: 25-40자 (한글 기준)
- 제목 스타일: 정보 제공형 + 숫자 활용
- 클릭을 유도하되 과장하지 않고 실용적으로
- SEO 키워드를 자연스럽게 포함
- 제목 바로 아래 두 번째 줄에 부제목 추가

절대 금지 표현:
- "~에 대해 알아보겠습니다"
- "~의 중요성은 두말할 필요가 없습니다"
- "결론적으로", "이처럼", "따라서"로 문단 시작
- 모든 소제목을 명사형으로 끝내기
- 숫자 나열 구조 남용 (3가지, 5가지 등)

행동 유도:
- CTA 스타일: {templateInfo.cta_style}
- SEO 전략: {templateInfo.seo_strategy}
```

## Claude 스타일 프롬프트
📍 위치: `src/Code.gs` / `generateStyleDescription()`

🎯 역할: 시트2의 숫자형 스타일 값을 사람이 이해할 수 있는 문체 지시문으로 바꿔 Claude system 프롬프트에 넣는다.

⚙️ 수정 가능 부분:

- 수치 구간 기준
- 각 구간 설명 문장
- 평균 문장 길이 표현 방식

⚠️ 주의사항:

- 숫자 기준을 바꾸면 기존 시트2 번호들의 성격이 한 번에 바뀐다.
- 문장을 너무 세세하게 규정하면 글이 딱딱해질 수 있다.

✏️ 수정 예시:

- 더 쉬운 글로 만들고 싶을 때
  - `expertise_level >= 8` 설명을 완화
- 더 간결한 글로 만들고 싶을 때
  - `sentence_style >= 8` 구간 문구를 짧게 변경

### 수치 항목 설명

- `writing_tone`: 글의 전문성, 격식, 거리감
- `sentence_style`: 문장 길이와 설명 밀도
- `personal_touch`: 개인 경험/주관 비중
- `visual_richness`: 목록, 불릿, 구조화 사용량
- `narrative_flow`: 스토리텔링 강도
- `expertise_level`: 전문 용어 수준
- `avg_sentence_len`: 평균 문장 길이 목표

### 현재 스타일 설명 전문

```text
적용할 스타일 특성 (스타일 번호: {number}):

[writing_tone]
- 8 이상: 매우 전문적이고 격식있는 톤으로 작성
- 6 이상: 전문적이면서도 접근하기 쉬운 톤으로 작성
- 6 미만: 친근하고 캐주얼한 톤으로 작성

[sentence_style]
- 8 이상: 상세하고 구체적인 설명 위주의 긴 문장 사용
- 6 이상: 적당한 길이의 문장으로 명확하게 설명
- 6 미만: 간결하고 명료한 짧은 문장 위주

[personal_touch]
- 8 이상: 개인적인 경험과 주관적 의견을 많이 포함
- 6 이상: 개인적 경험을 적절히 섞어 설명
- 6 미만: 객관적이고 사실 중심의 서술

[visual_richness]
- 6 이상: 불릿 포인트, 번호 목록 등을 적극 활용
- 6 미만: 연속적인 문단 형태로 자연스럽게 서술

[narrative_flow]
- 7 이상: 스토리텔링 형식으로 연결성 있게 서술
- 7 미만: 논리적 순서로 단계별 설명

[expertise_level]
- 8 이상: 전문 용어와 기술적 세부사항 포함
- 6 이상: 전문 지식을 쉽게 풀어서 설명
- 6 미만: 초보자도 이해하기 쉬운 용어 사용

[avg_sentence_len]
- 평균 문장 길이: 약 {avg_sentence_len}자 내외로 조절
```

### 수치 변경 시 예시

- `writing_tone` 낮춤
  - "해당 현상은 구조적 결함에 기인합니다."
  - → "이런 문제는 생각보다 자주 생깁니다."

- `sentence_style` 낮춤
  - 긴 설명형 문단
  - → 짧고 빠른 문장 위주

- `personal_touch` 높임
  - 객관식 설명
  - → "제가 현장에서 보면..." 같은 체감 표현 증가

## Gemini 나노바나나 이미지 프롬프트
📍 위치: `src/Code.gs` / `generateImagenPrompts_()`

🎯 역할: 블로그 본문에서 핵심 비교 요소와 자재명을 추출해, Gemini 이미지 생성용 프롬프트 2개를 JSON으로 생성한다.

⚙️ 수정 가능 부분:

- 비교형 이미지 템플릿 문장
- 단면 구조형 이미지 템플릿 문장
- Korean label 개수 제한
- 본문 잘라서 넣는 길이

⚠️ 주의사항:

- 출력 키 이름 `prompt1`, `prompt2`는 유지해야 한다.
- JSON 외 문장이 섞이면 후속 파싱이 깨진다.
- 회사명, 브랜드명, 지역명 금지 규칙은 유지하는 편이 안전하다.

✏️ 수정 예시:

- 비교형 이미지를 더 강한 광고형으로 바꾸기
  - `split-screen architectural technical infographic`
  - → `high contrast split-screen product comparison visual`

- 단면 이미지를 더 기술적으로 바꾸기
  - `heat flow arrows, moisture visualization`
  - → `layer callouts, performance annotations, airflow arrows`
