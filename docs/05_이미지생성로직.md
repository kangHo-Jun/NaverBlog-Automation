# 05_이미지생성로직

## 개요
- 현재 Naver 파이프라인은 본문 안의 사진 플레이스홀더와 보조 이미지 URL 기록을 함께 관리한다
- 이미지 관련 핵심 함수:
  - `processAutoImage_()`
  - `processUnsplashImage_()`
  - `processManualImage_()`
  - `mapPhotosToPlaceholders()`
  - `mapManualPhotosToPlaceholders_()`
  - `mapGeneratedImageUrlsToPlaceholders_()`
  - `extractPhotoPlaceholders_()`

## 사진 플레이스홀더 규칙
- 형식 예시:
  - `[사진 1: 슬림 창호 프레임 상세 사진]`
  - `[사진 2: 시공 전후 비교 사진]`
- 설명은 한글로 쓰는 것이 원칙
- 글 전체에 8~12개 수준이 권장
- 각 섹션과 자연스럽게 연결되어야 한다

## 이미지 소스 모드

### 자동생성
- `J열 = 자동생성`
- `runGenerateOnly()` 단계에서 image2 SVG URL을 먼저 생성해 `R열` 기록
- `generateOpenAIImage()`로 image1 생성 후 `M열` 기록
- `processAutoImage_()`는 `M열 image1_url`과 SVG image2를 사용해 플레이스홀더를 치환

### 직접업로드
- `J열 = 직접업로드`
- `runGenerateOnly()`가 글 제목 기준 Drive 이미지 폴더 생성
- `O/P/Q열`에 업로드 폴더, 필요사진수, 파일명가이드 기록
- `processManualImage_()`가 숫자 시작 파일명 기준으로 본문에 매핑

### Unsplash
- 코드에는 관련 함수가 남아 있다
- 현재 운영 문서 기준의 주력 모드는 `자동생성` 또는 `직접업로드`

## 파일명/검증 기준
- 숫자 시작 파일만 유효 파일로 인정
- 잘못된 파일명은 `getManualImageFileSummary_()`에서 걸러짐
- 플레이스홀더 수보다 유효 이미지 수가 적으면 후속 처리 중단 대상

## 실무 팁
- 자동생성은 빠르지만 프롬프트 품질 영향을 크게 받음
- 직접업로드는 시간이 들지만 결과 통제가 가장 좋음
- 플레이스홀더 설명을 구체적으로 써야 이미지 대응이 안정적이다
